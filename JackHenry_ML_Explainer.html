Feature Pipeline

Raw to Science Zone: Three Transformations

Bronze never changes. Silver unifies. The Feature Store anonymizes and versions.

BRONZE
Raw Ingest

Rail-native. Immutable. FI-encrypted.

  • NACHA, ISO 20022, Fedwire, RTP, FedNow stored exactly as received
  • Append-only with cryptographic hash on every record — no modification ever
  • Partitioned by rail_type / date / fi_tenant_id. FI-specific KMS keys.
  • No ML yet. Bronze is the audit layer and the source of truth for everything
SILVER
Unified Schema

18 rail types → 1 canonical schema. FI identifier intact.

  • Entity resolution maps originators to stable pseudonymous tokens across all rails
  • T₀–T₆ delta computation: 6 time intervals extracted per payment event
  • ACID merge/upsert handles late-arriving events (settlement confirmations up to 2 days late)
  • Last layer with FI attribution. Feeds both Gold (BI) and Feature Store (ML) — separate branches
FEATURE STORE
Science Layer

De-identified. DP-noised. Versioned. Model-ready.

  • Direct identifiers stripped. Pseudonymous tokens re-keyed. No FI label retained.
  • DP noise injected at boundary: ε ∈ [3, 8] per FI preference using DP-SGD batch noise
  • Output: row-level feature vectors (one per payment) — not aggregates, not summaries
  • Point-in-time versioned: every training run references a named, immutable dataset snapshot

Graph Intelligence

Three Feature Namespaces

The feature stack is organized into three namespaces — each capturing a different dimension of payment risk. Together, they give the model per-event, per-entity, and cross-network signal.

NAMESPACE 1

Per-Payment Temporal

Event-level features derived from timing, amount, and rail characteristics of each individual payment.

  • Δ₁–Δ₆ timing deltas between transaction lifecycle events
  • Amount bucket (logarithmic decile BUCKET_0–9)
  • Rail type one-hot (ACH, RTP, FedNow, Wire, Card)
  • Day-of-week and hour-of-day embeddings
  • SEC code classification (PPD, CCD, WEB, TEL)
  • Originator-relative amount Z-score vs 90-day history
Used by all 10 task heads. Purely event-local — no cross-entity lookup required.
NAMESPACE 2

Per-Entity Behavioral

Originator-level features capturing behavioral history — built from the entity's accumulated payment record.

  • 6-period rolling payment volume (weekly bins)
  • Rail diversity score (# distinct rails used at FI)
  • New counterparty introduction rate (past 30 days)
  • Amount bucket distribution shift (KL from baseline)
  • Time-since-last-payment (recency signal)
  • Counterparty reuse ratio (loyal vs. novel recipients)
Requires entity embedding lookup in Feature Store. Updated incrementally at each new event.
NAMESPACE 3

Consortium Network Signal

Cross-FI graph intelligence, pre-computed and bucketed centrally. No raw topology ever enters the Feature Store.

  • counterparty_prevalence_tier — how widely seen across consortium
  • originator_velocity_pct — decile of graph expansion rate
  • network_fraud_proximity — shortest path to confirmed fraud node
  • community_risk_tier — community-level historical fraud rate
  • cross_community_flag — originator and counterparty in different clusters
Privacy: Graph algorithms run centrally on de-identified Silver. Only bucketed enum scores enter Feature Store — no FI can reconstruct graph topology.
GRAPH CONSTRUCTION

Bipartite graph of originators → counterparties, built nightly from de-identified Silver. Edge weights = transaction frequency. Louvain algorithm detects communities (stable via token registry).

PRIVACY BY DESIGN

No entity ID, FI attribution, or raw amount traverses the consortium boundary. Pseudonymous stable tokens map real entities across runs without exposing identity. Only 5 integer/enum signals leave the graph compute layer.

WHY IT MATTERS

Fraud rings are network phenomena — they appear as dense communities with rapid counterparty expansion. NS3 lets every FI benefit from the consortium's collective fraud detection without sharing raw transaction data.

Compliance Intelligence

OFAC & Sanctions Screening

Three layers — fuzzy name normalization, entity embedding similarity, graph proximity — working together to cut false positives while catching risk the SDN list hasn't formalized yet.

The Gap in Traditional Screening:

Name-only OFAC screening compares raw strings against the SDN list. It misses transliterations (Arabic, Cyrillic, CJK script variants of the same name), aliases and AKAs in remarks fields, shell company ownership chains requiring the OFAC 50% rule, and unlisted entities whose behavioral fingerprint already matches sanctioned actors. Embedding + graph layers close all four gaps.

LAYER 1

Fuzzy Name Matching

Catch every orthographic variant before any embedding lookup.

  • Unicode normalization (NFC/NFD) + diacritic stripping
  • Transliteration: Arabic → Latin, Cyrillic → Latin, CJK → Pinyin (ISO 233 / BGN-PCGN)
  • SDN alias expansion — every AKA enters the candidate pool
  • Jaro-Winkler ≥ 0.80 → candidate set, top-20 per name token
  • Token-set ratio for multi-word entity names
  • Phonetic fallback: Double Metaphone for edge-case near-misses
Output: ranked candidate list with match_score 0–1 per SDN entry
LAYER 2

Entity Embedding Similarity

Behavioral fingerprint vs. the SDN entity embedding index.

  • Foundation model backbone produces 512-dim entity embedding from payment history
  • SDN entity index: pre-computed embeddings for all ~15K SDN entries + known proxies
  • FAISS IVF-PQ approximate nearest-neighbor — cosine similarity search in ~3–5ms
  • Captures behavior, not just name: velocity, rail mix, counterparty graph structure
  • An unlisted shell company that behaves like a sanctioned entity scores high here
  • SDN embedding index refreshed nightly as new designations are published
Output: top-5 SDN neighbors with cosine_sim 0–1 and embedding source
LAYER 3

Graph Proximity Score

Consortium network hop distance to confirmed sanctioned nodes.

  • SDN entities added as special nodes in the consortium transaction graph
  • BFS from target entity token to nearest SDN node — depth limit: 4 hops
  • OFAC 50% rule: traverse ownership edges — ≥50% owned by SDN node → auto-escalate
  • Shell company chain detection: 3–4 hop paths common in layered structures
  • Correspondent banking: wire counterparty tokens checked against SDN graph
  • Graph refreshed nightly — new confirmed sanctions trigger immediate re-index
Output: proximity_tier (DIRECT / ONE_HOP / TWO_HOP / DISTANT) + shortest path hop count

Model Reliability

Calibrated Scores = True Probabilities

Raw sigmoid output is not a probability. Platt scaling maps model scores to reliable risk estimates — enabling correct threshold-setting, cost-benefit analysis, and regulatory defensibility.

How Platt Scaling Works

1 · Pre-training
Foundation model trains with focal loss

Final layer applies sigmoid. Raw output f(x) ∈ [0,1] but is NOT a calibrated probability — the model is often overconfident in high-score buckets.

2 · Calibration Set
10–15% of confirmed FI labels held out

This data is NEVER used during training or fine-tuning. It exists solely to fit the calibration parameters A and B.

3 · Platt Fit
P(fraud) = σ(A · f(x) + B)

Fit logistic regression on the calibration set to minimize NLL. Result: any model score f(x) maps to a true posterior probability. Takes seconds — no retraining.

4 · Per-Head
Each task head gets its own A, B pair

ACH fraud, churn, liquidity — each has different class imbalance and score distribution. Shared calibration would be wrong.

5 · Platinum Tier
Per-FI recalibration from Platinum data

A large commercial FI may see 3× the base fraud rate of the consortium average. Platt parameters must reflect local base rates — not the consortium mean.

ECE (Expected Calibration Error) computed per head on each scoring cycle. Alert threshold: ECE > 0.03. Triggered recalibration runs automatically from latest Platinum-labeled data — no model retraining required.

Integration

API Surface

Four endpoints. Prediction and explanation are intentionally separate — only a fraction of predictions need to be explained. Label capture closes the learning loop.

Design principle: Explanation is decoupled from prediction. Real-time scoring skips SHAP computation entirely — saving ~2–4ms per call. Explanation is triggered on-demand when an analyst opens a case, or via policy (e.g., "always explain AUTO_HOLD decisions with score > 0.90").

LATENCY SPLIT

/predict: <20ms p99. /explain: ~80–150ms (SHAP is compute-heavy). Separation means the scoring path is never blocked by explanation computation.

LABEL ECONOMICS

Only ~3–8% of scored predictions are ever labeled (analyst reviews). /label is therefore low-volume but high-value — each confirmed label is worth months of model signal for small FIs.

TENANCY + ISOLATION

All API calls are scoped to fi_id. Platinum model weights, calibration parameters, and retention settings are isolated per FI. No cross-FI data leakage is possible at the API layer.
